# Scholarly Critique of StateCounter JOSS Paper

**Date**: January 19, 2026  
**Reviewer**: Claude Opus 4.5  
**Paper**: StateCounter: state tracking for composite objects in complex combinatorial libraries

---

## Overall Assessment

This is a well-written paper describing a useful and novel software contribution. The core concept—automatic state propagation through a DAG of composable counters—is clearly motivated and addresses a genuine gap. However, several aspects could be strengthened to better meet JOSS standards and maximize impact.

---

## Major Issues

### 1. Missing References (Critical for JOSS)

The paper cites PoolParty but includes no citations to external literature. JOSS papers should situate the software within a broader scholarly context. Consider adding:

- Citations for `itertools` and NumPy's `unravel_index` when discussing alternatives in "State of the Field"
- References to combinatorial testing literature (e.g., covering arrays, pairwise testing)
- A citation for directed acyclic graphs if you want to be precise about terminology
- Any published work on DNA library design that motivates the PoolParty application

The `paper.bib` file shows references to NumPy and pandas that are **never cited** in the paper text. Either cite them or remove them.

### 2. ORCID Placeholder

Line 11 has `orcid: 0000-0000-0000-0000` for Zhihan Liu—this needs to be replaced with a real ORCID or removed.

### 3. The "Statement of Need" Section Blurs into an Example

The Statement of Need section spends most of its text describing the specific DNA library use case (Fig. 1). While concrete motivation is valuable, JOSS expects this section to articulate the **general need** and **target audience** more explicitly. The general problem statement appears only in the final sentence. Consider restructuring to:
1. State the general problem first
2. Then give the DNA library as a motivating example
3. List other domains/applications explicitly (you mention them only in the final paragraph of the paper)

---

## Moderate Issues

### 4. Unclear Relationship to PoolParty

The paper mentions that StateCounter was "developed to support PoolParty" and is its "computational engine," but the relationship is unclear:
- Is StateCounter a standalone package or bundled with PoolParty?
- Is PoolParty published? (The bib entry says 2026—is it available now?)
- Why describe StateCounter separately rather than as part of PoolParty?

A brief clarification would help readers understand the software ecosystem.

### 5. "State of the Field" is Sparse

This section effectively describes what existing tools *don't* do, but JOSS reviewers often want to see:
- Any prior work that *partially* addresses this problem
- Related concepts from other domains (e.g., combinatorial test generation, configuration spaces)
- Why the problem hasn't been solved before (is this a new need, or an underappreciated one?)

The section is also quite short relative to the paper. Given the 1750-word limit, you might consider whether some of the code listing space could be better used here.

### 6. The Code Example Takes Significant Space

The code block (lines 68-94) and DAG output (lines 98-110) occupy ~30 lines. While illustrative, this could be trimmed:
- The `# Print sequences` line with `get_iteration_df()` is shown but never discussed
- The output DAG is helpful but could be referenced to the figure instead

Consider: Could Fig. 2 alone (with the state decomposition table) convey what readers need? The code could be moved to documentation.

---

## Minor Issues

### 7. Terminology Precision

- **"State propagation"**: The paper says "state information flows from root to leaves," but technically states propagate from derived counter to parent counters. The directionality language is confusing because "root" and "leaves" depend on how you view the DAG. Consider being more explicit: "When the user sets the state of a derived counter, that state propagates upstream to all ancestor counters."

- **"Counter algebra"**: This term appears in tags and README but not in the paper body. If it's a key concept, introduce it.

### 8. Figure Captions

- **Fig. 1 caption**: At 72 words, this is quite long and includes technical details that might fit better in the text. The caption also introduces the term "v0 and v1" for barcodes without explaining that `v_counter` has 2 states.

- **Fig. 2**: The caption is clearer, but the figure itself (not shown to me as an image, but described) seems dense. Ensure it's legible at typical PDF sizes.

### 9. Operations Table Completeness

The operations table lists 10 operations, but the code example only demonstrates `product` and `stack`. Consider whether the table needs all 10 listed, or whether a reference to full documentation would suffice. Some operations (`passthrough`) seem minor.

### 10. AI Usage Disclosure

The disclosure is commendable and increasingly expected, but "Z.H." appears once (should this be "Z.L."?).

## Positive Aspects to Preserve

1. **Clear problem articulation**: The opening example of "given value $n$, how do we compute which mutations to introduce" is compelling.

2. **Good figures**: Fig. 1 shows a concrete application, and Fig. 2 demonstrates the core value proposition (state decomposition).

3. **Conflict detection**: The discussion of how the DAG can share counters and how conflicts are detected is technically interesting and differentiates StateCounter from simpler approaches.

4. **Honest AI disclosure**: This sets a good example.

5. **Clean writing**: The prose is clear and professional.
